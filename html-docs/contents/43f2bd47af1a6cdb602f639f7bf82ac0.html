Description

This use case describes different actions that a user (Tourist) can perform by interacting with TRS GUI (Travel Recommender System GUI). These actions can be grouped into to main phases:

1) Itinerary Request. The user queries the system to get personalized itineraries, based on her preferences (level of interest in the different TOIs - Topic of Interest) and on her time and mobility constraints. The specification of these information is therefore mandatory for the user. Optionally, the user can authenticate to the system in order to save and retrieve her data throughout multiple navigation sessions.

2) View Itineraries. Once the system has returned a set of personalized itineraries, the user can get a summary of them in a succinct way (e.g., as a list with a brief description of each itinerary), and then explore each of these itineraries in more detail (e.g., displaying the sequences of POIs traversed by the itinerary and some information associated with each POI). Optionally the user can: (i) post a feedback about one or more itineraries; (ii) explore an itinerary with extended reality, if there are available extended reality contents for the selected itinerary and provided that the user has the right devices; (iii) export an itinerary in some predefined formats (such as the KML Google Earth format) or save it in her account, provided that the user has previously authenticated to the system.
